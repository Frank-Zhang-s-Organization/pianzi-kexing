# 维护者个人信息去风险化策略

## 现状判断

你当前的公开暴露面已经足以支撑定向骚扰：

- GitHub 公开账号可见用户名、昵称、所在地、技术兴趣和个人站仓库。
- 个人站源码里公开了直连邮箱。
- 页面模板里预留了微信二维码展示入口。
- About 页结构化数据还会把邮箱写入 `Person` JSON-LD，方便被搜索引擎和爬虫直接抽取。

这意味着攻击者不需要复杂检索，只要把 GitHub 账号、个人站、邮箱和社交入口串起来，就能做：

- 邮件轰炸
- 短信或电话社工扩展
- 社交平台定向骚扰
- 基于地域和长期兴趣的开盒拼图

## 风险分级

### P0: 上线前必须处理

- 从公开页面移除直连邮箱，改为表单或一次性别名邮箱。
- 移除微信二维码和任何可被批量加好友的入口。
- GitHub 个人资料里的所在地改为更宽泛区域，不展示到城市级。
- 新项目不要从个人账号首发，改用 GitHub Organization。

### P1: 一周内完成

- 为开源项目单独准备维护者邮箱别名，例如 `oss@...`，并接入垃圾过滤。
- 域名开启 WHOIS 隐私保护。
- 所有通知渠道启用 2FA，GitHub 开启 passkey 或安全密钥。
- 准备一套公开沟通模板，避免在 issue / 邮件里暴露更多个人信息。

### P2: 被针对时立即启用

- 邮箱层面开启更严格的收件规则和黑名单。
- 手机号不再绑定公开可见服务，必要时换成单独工作号。
- 关闭仓库 Discussions，限制 issue 模板字段，启用协作守则和举报入口。
- 明确“只接受邮件别名 / issue / 表单”三种沟通入口，收缩战线。

## 推荐发布架构

### 身份层

- 个人账号只保留普通开发活动，不作为项目门面。
- 新建 GitHub Organization 承载仓库、文档和 releases。
- 项目 README、网站和 issue 模板中不出现真实邮箱、微信、手机号。

### 联系层

- 对外只保留一个表单或邮件别名。
- 不公开任何即时通信二维码。
- 所有通知经过邮件规则、Cloudflare Email Routing 或其他中转层。

### 仓库层

- 关闭不必要的社交暴露功能。
- 默认使用 issue 模板，让外部输入结构化。
- 提前写好 `SECURITY.md`、`CODE_OF_CONDUCT.md`、`SUPPORT.md`。

## 建议的执行顺序

1. 先处理个人站里的邮箱、微信二维码和结构化数据。
2. 再调整 GitHub Profile 的位置、简介和对外联系方式。
3. 最后再把项目放到组织账号中公开。

## 不建议做的事

- 不要把“防止开盒”寄希望于把邮箱写成图片。
- 不要继续把个人站、个人邮箱、微信二维码和高争议项目绑在同一个身份上。
- 不要用个人手机号注册对外联络入口。
